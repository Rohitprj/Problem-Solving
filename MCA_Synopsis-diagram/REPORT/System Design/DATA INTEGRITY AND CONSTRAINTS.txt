DATA INTEGRITY AND CONSTRAINTS

1. INTRODUCTION

Data is one of the most valuable assets of any software system. If the stored data is inaccurate, duplicated, incomplete, inconsistent, or unauthorized, the entire system can fail to provide correct results. Therefore, maintaining data integrity is a critical part of system design. Data integrity means ensuring that data remains accurate, valid, consistent, complete, and reliable throughout its lifecycle—from data entry to storage, updates, retrieval, and deletion. Database systems commonly enforce integrity through rules known as constraints. These rules restrict invalid operations and help preserve quality of data.

In the Spiritual AI Chat Bot project, user records, login credentials, chat history, analytics, admin data, and knowledge base content are continuously stored and accessed. If data integrity is not maintained, problems such as duplicate accounts, broken chat history, unauthorized access, wrong reports, or inconsistent responses may occur. Hence, strong data integrity mechanisms and constraints are necessary.

4. TYPES OF INTEGRITY CONSTRAINTS APPLIED

Database systems often use several major categories of integrity constraints such as entity integrity, referential integrity, domain integrity, and key constraints.

4.1 Entity Integrity

Entity integrity means each table/document collection must uniquely identify every record. In relational systems, this is commonly enforced through a Primary Key, which must be unique and not null.

Application in This Project:

User Collection:
4.1.1 userId – unique database identifier  

Chat Collection:
4.1.2 chatId – message identifier  

Admin Logs:
4.1.3 logId  

This ensures every record can be uniquely accessed.

4.2 Referential Integrity

Referential integrity ensures that relationships between records remain valid. A foreign key or reference must point to an existing parent record.

Application in This Project:

Examples:
4.2.1 Chat history must reference an existing user  
4.2.2 Admin reports must reference valid users  
4.2.3 Saved favorites must belong to valid accounts  

If a user account is deleted, related chat records should be handled using:
4.2.4 Cascade delete  
4.2.5 Archive policy  
4.2.6 Soft delete approach  

This prevents orphan records.

4.3 Domain Integrity

Domain integrity ensures that values stored in fields match the allowed type, range, and format.

Application in This Project:

Email Field:
4.3.1 Must follow valid email format  

Password Field:
4.3.2 Minimum length and complexity  

Role Field:
4.3.3 Allowed values only: USER, ADMIN, SUBADMIN  

Message Length:
4.3.4 Text messages should remain within defined limits  

Rating Field:
4.3.5 Feedback score allowed only between 1 and 5  

4.4 Key Constraints

Key constraints prevent duplicate values in important fields such as usernames or emails.

Application in This Project:

4.4.1 Email Address – must be unique  
4.4.2 Mobile Number (if used) – must be unique  
4.4.3 Username (if enabled) – must be unique  
4.4.4 Admin ID – must be unique  

This avoids duplicate accounts and confusion.

5. COMMON CONSTRAINTS USED IN THE PROJECT

5.1 NOT NULL Constraint

Some fields must never remain empty.

Examples:
5.1.1 Name  
5.1.2 Email  
5.1.3 Password  
5.1.4 Role  
5.1.5 Created Date  

5.2 UNIQUE Constraint

Ensures duplicate values are not inserted.

Examples:
5.2.1 Email  
5.2.2 Username  
5.2.3 Admin Login ID  

5.3 PRIMARY KEY Constraint

Uniquely identifies each record.

Examples:
5.3.1 userId  
5.3.2 chatId  
5.3.3 reportId  

5.4 FOREIGN KEY / REFERENCE Constraint

Used to link related data.

Examples:
5.4.1 chat.userId references user.userId  
5.4.2 feedback.userId references user.userId  

5.5 CHECK Constraint

Used to restrict values.

Examples:
5.5.1 Age > 13  
5.5.2 Rating between 1 and 5  
5.5.3 Role in approved list  

5.6 DEFAULT Constraint

Provides default values automatically.

Examples:
5.6.1 status = active  
5.6.2 role = user  
5.6.3 createdAt = current date  
5.6.4 isBlocked = false
